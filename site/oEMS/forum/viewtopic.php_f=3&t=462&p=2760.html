It depends on the machine used, and the Linux compilation as well. On the 4 processor machine, there is a slight increase in speed, from 77.1 MC/s to 77.7 MC/s.

The fftw function is open, under gnu license, and can be used to set the system-wide fft routine.

http://www.fftw.org/

It appears that Octave has included all or part of this code.

I believe that subsequent calls, from any routine, including C++ that you have written, for fft or ifft will use the routine which fftw finds and stores in a file somewhere, although I haven't figured out where on Linux. The Octave manual claims that for Unix, it is stored in /etc/fftw/wisdom.

It appears that if your code calls fft or ifft, which I presume it does, it will use the routine which fftw (for fft wisdom) has selected. It does not have to be called from octave.

Clearly, this line of code is not of much use on all machines, but, for those Unix or Linux users who want to try it, it appears harmless, and resulted in considerable improvement in simulation time on one of my Linux machines.

Post by thorsten » Sun 07 May 2017, 08:56

I can only repeat, something else must be causing what you observe.
The FDTD engine does not call any external functions. openEMS does not use any kind of fft or ifft anywhere.
The iteration engine is not doing anything else than add and multiply values... This just CANNOT be the cause.

Maybe your mesh and general setup is changed by Octave somehow and this is causing the speed difference. Do you use field dumps? Try disabling them.
Please try running openEMS without Octave and see which speed you see.
On the 4 processor machine, there is a slight increase in speed
More cores do not necessarily help much. It is the memory (and L2 cache) bandwidth that really matters.
The CPU itself usually has enough reserves for 10-100x faster simulation. The limit is the data bandwidth. With other words, the data just does not get fast enough to the CPU, the CPU is mostly waiting for new numbers to crunch....
But this means as well that if Octave (or anything else) is doing something funny in the background it may disturb the memory data flow...
