Hi Olf,

maybe I have a very simple way, but maybe it may not work at all...

I modified the openEMS.sh script to this:

Code: Select all

#!/bin/bash

#get path to openEMS
openEMS_PATH=`dirname $0`

#set LD_LIBRARY_PATH to the path of this script
export LD_LIBRARY_PATH=$openEMS_PATH

#execute openEMS
exec $openEMS_PATH/openEMS $@
And put all necessary libs and executables into the folder. Thereby openEMS can run using (mostly) all libs from there...

I created a tar.bz containing my current (standard) build with all necessary libs (from my Ubuntu 12.04) and put it here: http://openems.de/download/linux/openEM ... -11.tar.gz
I was able to run this on a "Live CD" of Ubuntu 12.10 ... but I don't know about e.g. Suse or other systems...

Advantage (if it works) you can just take the "standard" openEMS build, extract all necessary libs from your system and put it all in one folder and run it on a completely different machine...

Maybe you can give this a try on your cluster...

Hi Olf,

thanks for the feedback (twice )

To sum it up, you had all this trouble because you couldn't just use the dev packages of hdf5, tinyxml, fparser, vtk and boost from your linux distro because you didn't have root access to install them right?
Or the really most simple way for Linux user, use the openEMS repository and be finished within a minute But again that requires root access to install the packages into the system...
I changed the number of threads by executing openEMS with " --engine=multithreaded --numThreads=8" and it runs much faster than the standard execution.
You can just use "--numThreads=8" the multithreaded engine is he default anyways. And yes I noticed that just using the max number of available threads isn't always the best choice. But there is no easy way to find the best setup.. I was thinking about some automatic adaption during run-time (for some version in the future)...

Code: Select all

If I want to use mpi do I just need to start openEMS with mpirun?
There is a Matlab function called RunOpenEMS_MPI have a look into the help function. At least you need SetupMPI in Matlab to tell openEMS how to split up the simulation.

But keep in mind the current release version (v0.0.29) has a serious "access after delete" bug killing any MPI enabled simulation... You will need the latest dev version. I will try to push it to github today.
And still, MPI is highly untested and very unstable... Since I use it nearly never and nobody else does there are very likely a lot more bugs, especially since debugging MPI is really difficult...
Furthermore not all features are yet compatible with MPI openEMS, e.g. the total-field/scattered-field approach. And also nf2ff and field dumps in general are more difficult to handle with MPI...

So I suggest you only use MPI if you really really have to, e.g. the simulation gets to large for one node.
Still I would be very curious about the performance on a linux cluster. I can only try it on a cluster of individual PC's with a rather slow network connection...

Hi Thorsten,
I didn't run into memory problems yet so I will just keep to the multithreaded non-mpi version which works excellent for me now.
The installation of the dependencies turned out to be one problem, the compiling of the source was the second one.

For the dependencies, I have the following comments/remarks:

- boost needs to be built, not just copied as it is said on the boost website (some of the boost libraries that openEMS uses are not header only).
The installing routine that worked for me goes as follows
a. Download and extract
b. ‘./bootstrap.sh --prefix=$HOME/install ’
c. Add the line ‘using mpi;’ to project-config.jam
d. ‘b2 install --build-type=complete --layout=tagged –prefix=$HOME/install’
e. Copy libraries and headers to $HOME/install (somehow it didn’t do it automatically)

- if you want to compile hdf5 yourself you need to set the variable --enable-cxx for the configure script (I settled with the precompiled version from their website)

- the other dependencies were more or less straight forward

For installing and compiling the openEMS code I had to tinker a bit with the script and the .pro files

- For some reason there were some linking problems, I added -L/<lib path> to the lines in the .pro files where it says 'LIBS+= ...'

- the variable GIT_VERSION was not defined, so I had to comment out the lines where it is used (maye this is because I didn't download the source with git?)
- another variable called AUTHOR (or OPENEMS_AUTHOR?, forgot how it was called exactly) was also not defined, solved the issue the same way

- had to add -fPIC flag to the make command, that is what the compiler told me to do

The install script that works for me now looks like:

Code: Select all

#!/bin/bash
for mod in CSXCAD QCSXCAD AppCSXCAD ; do

        echo "Pull $mod from server & rebuild"
        cd $mod

        #git pull origin master
        #if [ $? -ne 0 ]; then
        #  echo "Pulling $mod failed"
        #  exit
        #fi

        make clean &> /dev/null

        $HOME/install/bin/qmake
        #if [ $? -ne 0 ]; then
        #  echo "qmake for $mod failed"
        #  exit
        #fi

        echo "compiling $mod ... please wait"
        make CXXFLAGS="-fPIC -L$HOME/install/lib -L$HOME/install/lib/vtk5.10 -I$HOME/install/include -I/home/hudelistf/install/include/vtk-5.10" > /dev/null
        if [ $? -ne 0 ]; then
          echo "make for $mod failed"
          exit
        fi

        cd ..
done

cd openEMS
make clean &> /dev/null
qmake
echo "compiling openEMS ... please wait"
make -j4 CXXFLAGS="-DH5_USE_16_API -L$HOME/install/lib -L$HOME/install/lib/vtk5.10 -I$HOME/install/include -I/home/hudelistf/install/include/vtk-5.10" > /dev/null
make -j4 > /dev/null
if [ $? -ne 0 ]; then
  echo "make for openEMS failed"
  exit
fi
cd ..



cd openEMS/nf2ff
make clean &> /dev/null
qmake
echo "compiling nf2ff ... please wait"
make -j4 CXXFLAGS="-I$HOME/install/boost/boost""> /dev/null
if [ $? -ne 0 ]; then
  echo "make for nf2ff failed"
  exit
fi
cd ../..

if which octave &> /dev/null;
then
 echo "octave found, running setup"
 octave -q --eval "addpath('$PWD/CSXCAD/matlab');addpath('$PWD/openEMS/matlab');setup;exit;"
fi

echo "openEMS and all modules have been updated successfully... Have fun using openEMS"
The addpath command didn't work, but I could add the paths manually to octave later. Also some of the changes might not be necessary, I just kept changing things until everything worked.

Hope that is some helpful feedback for you!
